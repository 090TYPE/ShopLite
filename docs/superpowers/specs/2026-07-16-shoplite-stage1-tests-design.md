# ShopLite Этап 1 — доменный слой и тесты

Дата: 2026-07-16
Статус: утверждено, готово к планированию реализации

## Задача

Довести ShopLite до состояния, в котором тестовая пирамида репозитория выдерживает
senior-ревью: доменные правила покрыты unit-тестами, каждый API-эндпоинт имеет
интеграционный тест на реальном PostgreSQL, и один E2E-сценарий доказывает, что
событие реально доезжает через брокер до другого сервиса.

Этап 1 не трогает Outbox, наблюдаемость, Kubernetes и CI — это Этапы 2–5,
каждый отдельным циклом.

## Исходное состояние

Разведка репозитория (коммит `55cbaa5`) показала:

- `Order` — анемичная модель: только автосвойства, ни одного инварианта.
- Единственный расчёт — `order.Total = order.Items.Sum(i => i.Price * i.Quantity)`
  строкой внутри `OrdersController.Create`.
- Валидации нет: пустой `Items`, `Quantity = -5`, отрицательный `Price` создают
  заказ с бессмысленной суммой. Это действующий баг, а не гипотетический.
- `GenerateJwt` — приватный метод контроллера, недоступный тесту.
- Контроллеры зависят от `OrderDbContext` и `IPublishEndpoint` напрямую.

Следствие: цель «60–70% покрытия домена» недостижима не из-за нехватки тестов, а
потому что домена нет. Unit-тесты по текущему коду проверяли бы автосвойства или
мокали `DbSet` — оба варианта на ревью работают против репозитория. Поэтому
Этап 1 начинается с извлечения домена.

## Решение

### Доменный слой

Живёт в `OrderService/Domain/`. Отдельная сборка не заводится — для одного
агрегата это лишняя церемония (YAGNI).

**`Order.Create(userId, items)`** — единственная точка рождения заказа.

Правила:

- `Items` непустой;
- `Quantity > 0` для каждой позиции;
- `Price >= 0` для каждой позиции;
- `ProductName` не пустой и не whitespace;
- `Total` вычисляется внутри фабрики; снаружи не присваивается.

Сеттеры агрегата закрываются до `private set`, чтобы обойти фабрику было нельзя.

**Переходы статусов:** `MarkProcessing()`, `MarkShipped()`, `Cancel()` с
проверкой допустимости — нельзя отгрузить отменённый, нельзя отменить
доставленный. Эти же переходы станут основой саги в Этапе 2.

### Обработка ошибок

Валидация возвращает `Result<Order>`, не бросает исключения.

Обоснование: невалидный ввод от клиента — ожидаемый сценарий, а не
исключительный. На исключениях интеграционные тесты 400-ок вырождаются в
проверку middleware, ловящего `DomainException`, что смешивает предмет теста с
транспортом.

Контроллер маппит `Result.Failure` в `400 ValidationProblem`. Тело
`Create` сокращается до вызова фабрики, `SaveChangesAsync` и `Publish`.

### UserService

Домена почти нет. Извлекается `JwtTokenGenerator` за интерфейс — сейчас это
приватный метод контроллера, поэтому непроверяем. Даёт предмет для
`UserService.UnitTests` и переиспользуется в E2E.

## Структура тестов

Пять проектов, как в исходном плане — границы жёсткие, каждый проект тянет
только свои пакеты, структура читается ревьюером с первого взгляда.

```
tests/
  UserService.UnitTests/
  UserService.IntegrationTests/
  OrderService.UnitTests/
  OrderService.IntegrationTests/
  ShopLite.E2ETests/
```

| Проект | Предмет проверки |
|---|---|
| `OrderService.UnitTests` | правила `Order.Create`, переходы статусов — без инфраструктуры |
| `UserService.UnitTests` | `JwtTokenGenerator` (claims, срок, подпись), хеш пароля |
| `OrderService.IntegrationTests` | Testcontainers Postgres + `WebApplicationFactory`; POST /orders → 201 → строка в БД; 400 на невалидном; 401 без токена; чужой заказ → 404 |
| `UserService.IntegrationTests` | register → 201; дубль email → 409; login → токен; неверный пароль → 401 |
| `ShopLite.E2ETests` | register → login → order → `OrderCreated` доехал до `NotificationService` |

### Брокер в тестах

Интеграционные тесты используют `MassTransit.TestFramework` (in-memory harness):
контейнер Rabbit там не нужен и только замедляет прогон.

Реальный `Testcontainers.RabbitMq` поднимается только в E2E — именно там
предметом проверки является настоящая доставка через брокер.

### Жизненный цикл контейнеров

Контейнер поднимается один раз на сборку через xunit collection fixture; между
тестами чистится база. Контейнер на класс при пяти проектах сделает прогон
невыносимым.

## Пакеты

`xunit`, `FluentAssertions`, `NSubstitute`, `Testcontainers.PostgreSql`,
`Testcontainers.RabbitMq`, `Microsoft.AspNetCore.Mvc.Testing`,
`MassTransit.TestFramework`, `coverlet.collector`.

## Поправка от 2026-07-16: репозиторий переведён на .NET 10

Раздел ниже написан до начала реализации и на момент утверждения был верен. При
выполнении задачи 2 выяснилось большее: рантайма .NET 9 на машине нет вообще —
`dotnet --list-runtimes` показывает только 8.0.27, 8.0.28 и 10.0.8. Сборка
проходила (SDK 10 компилирует под net9.0 по reference-пакетам из NuGet), но всё
исполняемое падало: `Framework: 'Microsoft.NETCore.App', version '9.0.0' not
found`. Репозиторий никогда и не запускался на хосте — только через
docker-compose, где рантайм несут образы. Этап 1 весь про тесты, которым нужен
именно хост.

Вдобавок .NET 9 — STS-релиз, поддержка которого закончилась в мае 2026, за два
месяца до этих работ. Портфельный репозиторий на снятом с поддержки фреймворке —
плохой сигнал.

Решение утверждено пользователем: репозиторий переведён на .NET 10 (LTS,
поддержка до ноября 2028) коммитом `5cc5272`. Все проекты — `net10.0`,
Microsoft-пакеты — 10.x, Npgsql — 10.0.3, Dockerfile'ы — `sdk:10.0` и
`aspnet:10.0` (у NotificationService — `runtime:10.0`: это воркер без HTTP).
MassTransit остался на 8.3.5, Swashbuckle — на 6.9.0: оба собираются под net10.0
без правок. FluentAssertions остаётся на 7.2.0 по лицензионным причинам.

Читать раздел ниже как исторический.

## Известные ограничения окружения (на момент утверждения спеки)

- `Program.cs` вызывает `EnsureCreated()`. Фабрика тестов подменяет регистрацию
  `DbContext` на строку подключения от Testcontainers. В Этапе 2 это всё равно
  придётся заменить на миграции — Outbox от MassTransit их требует.
- На машине разработки нет .NET 9 SDK; сборка идёт на SDK 10.0.300 и проходит
  чисто. В `ci.yml` Этапа 5 нужно явно ставить `9.0.x` через `setup-dotnet`.
- В соседнем репозитории CryptoAI лежит `global.json`, пиннящий SDK 8. Команды
  `dotnet` для ShopLite нужно запускать из каталога ShopLite, иначе резолвится
  чужой SDK и сборка падает с `NETSDK1045`.

## Порядок работ

1. Домен по TDD: тесты на правила — красные, затем реализация.
2. Перевод `OrdersController` на домен.
3. Извлечение `JwtTokenGenerator` + его тесты.
4. Интеграционные тесты обоих сервисов.
5. E2E-сценарий.
6. `coverlet.collector` и отчёт покрытия.

## Критерии готовности

- `dotnet test` зелёный из каталога ShopLite.
- Каждый эндпоинт обоих API имеет минимум один интеграционный тест.
- Покрытие домена 60–70%.
- Заказ с пустыми позициями, нулевым количеством или отрицательной ценой
  отклоняется с 400 — подтверждено тестом.
